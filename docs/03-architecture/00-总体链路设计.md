# 总体链路设计

## 1. 文档目的

本文档用于定义新能源标定系统的总体链路结构、分层职责、关键对象和闭环关系，作为后续模块设计与三层建模的上位输入。

## 2. 适用范围

适用于当前项目的：

- 总体数据链路设计
- 总体决策链路设计
- 模块职责划分
- 模型分层边界定义

不直接覆盖：

- 具体模块字段设计
- 具体物理表设计
- 具体算法规则表达

## 3. 架构目标

本架构设计的目标如下：

- 统一承载外部数据与内部数据
- 形成可追溯、可计算、可闭环的数据链路
- 通过 `TestContext` 形成统一决策输入
- 通过 `Event` 和 `Action` 让过程具备结构化表达能力
- 通过 `Result` 和 `Feature` 沉淀长期数据资产

## 4. 分层结构

### 4.1 数据产生端

职责：

- 产生原始测量数据
- 产生设备反馈数据
- 产生外部系统输入数据
- 产生系统内部动作与执行反馈

输入：

- 设备运行状态
- 外部采集系统输出
- 控制执行结果

输出：

- 原始外部数据
- 原始内部数据

### 4.2 数据接入层

职责：

- 接收不同来源的输入数据
- 保留来源信息和接入时间信息
- 完成接入层级的数据落点管理

输入：

- 外部原始数据
- 内部原始数据

输出：

- 已接入原始数据

### 4.3 数据标准层

职责：

- 统一数据格式
- 统一时间语义
- 统一编码体系
- 统一字段结构

输入：

- 已接入原始数据

输出：

- 标准数据
- 标准化上下文数据

### 4.4 数据融合层

职责：

- 组合多源标准数据
- 组合上下文信息
- 形成统一业务对象 `TestContext`

输入：

- 标准数据
- 上下文数据
- 当前控制状态

输出：

- `TestContext`

### 4.5 数据计算层

职责：

- 基于 `TestContext` 进行状态判断
- 基于时间窗口进行特征计算
- 输出业务判断结果

输入：

- `TestContext`

输出：

- 计算结论
- 稳定性判断
- 特征数据

### 4.6 事件与动作层

职责：

- 将关键状态变化结构化为 `Event`
- 将控制决策结构化为 `Action`
- 记录动作执行反馈

输入：

- 计算结论
- 决策结果
- 执行反馈

输出：

- `Event`
- `Action`

### 4.7 结果层

职责：

- 沉淀阶段性结果
- 沉淀执行结果
- 沉淀分析结果与训练样本

输入：

- `Event`
- `Action`
- 计算结论
- 特征数据

输出：

- `Result`
- `Feature`

### 4.8 数据存储层

职责：

- 存储原始数据、标准数据、上下文数据、事件、动作、结果、特征
- 提供可追溯的历史数据基础

输入：

- 各上层输出数据对象

输出：

- 可查询、可追溯的数据资产

### 4.9 数据服务层

职责：

- 向上层应用提供数据访问服务
- 向分析、评审、算法、外部系统提供数据查询能力

输入：

- 已存储数据资产

输出：

- 查询结果
- 服务化数据对象

### 4.10 数据管理层

职责：

- 维护数据标准
- 维护数据质量规则
- 维护元数据、字典、版本和治理规则

输入：

- 全链路数据对象和治理规则

输出：

- 数据治理结果
- 质量约束
- 口径定义

## 5. 关键对象

| 对象名称 | 对象类型 | 职责 |
| --- | --- | --- |
| Raw Data | 数据角色 | 保留原始输入形态，用于追溯和重放 |
| Standard Data | 数据角色 | 形成统一结构和统一编码后的数据对象 |
| Context Data | 数据角色 | 提供任务、工况、变量、控制模式等业务语义 |
| TestContext | 业务对象 | 多源融合后的统一输入核心对象 |
| Event | 业务对象 | 记录关键状态变化 |
| Action | 业务对象 | 表达系统动作和控制意图 |
| Result | 业务对象 | 沉淀阶段性业务结果 |
| Feature | 业务对象 | 沉淀用于训练和优化的特征数据 |

## 6. 关键链路

### 6.1 数据主链路

数据产生端
-> 数据接入层
-> 数据标准层
-> 数据融合层
-> 数据计算层
-> 结果层
-> 数据存储层
-> 数据服务层

### 6.2 决策主链路

`TestContext`
-> 状态判断
-> 决策输出
-> `Action`
-> 执行动作
-> 反馈数据回流

### 6.3 结构化过程链路

连续数据
-> 状态判断
-> `Event`
-> 阶段结果
-> 数据沉淀

### 6.4 闭环链路

数据
-> `TestContext`
-> 判断
-> 决策
-> `Action`
-> 执行
-> 反馈数据
-> 数据

## 7. 边界与依赖

### 7.1 外部边界

外部边界包括：

- 测量设备
- 试验台架
- 外部控制系统
- 外部任务系统

### 7.2 内部边界

内部边界包括：

- 数据接入与标准化边界
- 数据融合与决策边界
- 事件动作与结果沉淀边界
- 数据存储与服务边界

### 7.3 依赖关系

总体依赖顺序如下：

1. 数据接入层依赖数据产生端
2. 数据标准层依赖数据接入层
3. 数据融合层依赖标准数据与上下文数据
4. 数据计算层依赖 `TestContext`
5. 事件与动作层依赖计算结果与决策结果
6. 结果层依赖事件、动作、计算结果和特征
7. 数据服务层依赖数据存储层
8. 数据管理层横向作用于全链路

## 8. 约束

总体架构必须持续遵守以下约束：

- `TestContext` 是统一直接输入核心对象
- 决策链路不得绕过数据融合层直接消费原始数据
- 事件必须表达关键状态变化，不得沦为普通日志替代物
- 动作必须作为正式数据对象纳入链路，不得只保留执行痕迹
- 结果层只沉淀具有明确业务价值的产物
- 各层只表达本层职责，不跨层承载语义

## 9. 待确认事项

- 外部系统边界清单是否需要单独拆分
- 数据管理层是否单独作为治理域，而不是业务链路层
- `Feature` 是否全部进入结果层统一沉淀，还是保留独立特征子层
- 决策链路是否需要单独形成策略层与执行层两级结构
