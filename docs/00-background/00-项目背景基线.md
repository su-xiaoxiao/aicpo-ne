# 新能源标定系统背景基线

## 1. 项目定位

本项目面向新能源/发动机标定场景，目标是构建一套以数据驱动为核心的工业闭环系统。

系统不是单一的数据接入平台，而是覆盖以下完整链路的数据工程与决策驱动系统：

数据采集 -> 数据处理 -> 状态判断 -> 控制决策 -> 执行动作 -> 数据反馈 -> 结果沉淀 -> 策略优化

系统需要统一处理两类数据：

- 外部数据：测量数据、设备反馈、外部系统输入
- 内部数据：控制动作、上下文、事件、结果

项目本质定义为：

一个完整的数据工程 + 决策驱动 + 闭环回流的工业数据系统

---

## 2. 系统目标

### 2.1 数据统一

统一纳管外部数据与内部数据，形成一条可追踪、可计算、可回流的统一数据链路。

统一处理对象包括：

- 外部测量数据
- 设备反馈数据
- 外部系统数据
- 控制动作数据
- 运行上下文数据
- 状态事件数据
- 结果沉淀数据

### 2.2 多源融合

通过融合层构建统一业务对象 `TestContext`。

`TestContext` 是系统核心输入对象，用于承载：

- 实时测量
- 当前上下文
- 当前控制状态
- 当前时间窗口

后续判断、计算、决策均以 `TestContext` 作为统一输入。

### 2.3 过程结构化

通过 `Event` 描述过程中的关键状态变化，使连续试验过程具备可解释的结构化表达能力。

典型事件包括：

- `STABLE_REACHED`
- `RECORD_TRIGGERED`
- `CONTROL_SENT`
- 工况切换事件
- 异常事件

### 2.4 闭环驱动

系统必须支持完整闭环：

数据 -> 判断 -> 决策 -> 动作 -> 数据

该闭环用于支撑：

- 自动标定
- 策略优化
- 模型训练

### 2.5 数据资产沉淀

通过结果层和特征层沉淀试验过程中的关键业务产物，形成长期可复用的数据资产。

沉淀对象包括：

- 试验结果
- 执行结果
- 特征数据
- 可用于建模和优化的样本

---

## 3. 总体链路

系统采用统一数据链路结构：

数据产生端
-> 数据接入层
-> 数据标准层
-> 数据融合层
-> 数据计算层
-> 事件与动作层
-> 结果层
-> 数据存储层
-> 数据服务层
-> 数据管理层

该链路是后续模块设计、数据建模和职责拆分的总体主线。

---

## 4. 两条核心链路

### 4.1 数据链路

数据链路负责：

- 数据接入
- 数据标准化
- 数据融合
- 数据计算
- 数据存储
- 数据服务

### 4.2 决策链路

决策链路负责：

- 基于 `TestContext` 进行状态判断
- 基于判断结果生成 `ControlAction`
- 驱动控制动作执行

### 4.3 链路关系

两条链路的衔接关系定义为：

数据 -> TestContext -> 决策 -> Action -> 数据

其中：

- 数据链路负责提供可计算、可追溯的数据基础
- 决策链路负责消费融合结果并输出动作
- 动作执行后的反馈数据重新进入数据链路，形成闭环

---

## 5. 核心数据角色

系统中的数据统一划分为以下角色。

### 5.1 Raw Data

原始数据，来源于外部输入或系统内部原始输出，保持原始形态，主要用于追溯、审计和重放。

### 5.2 Standard Data

标准数据，指原始数据经过结构统一、编码统一和格式标准化后的数据对象。

### 5.3 Context Data

上下文数据，用于提供数据解释和业务语义，包括但不限于：

- 当前任务
- 当前工况
- 当前变量定义
- 当前控制模式

### 5.4 Business Object

融合对象，表示跨来源、跨时间窗口、跨语义的数据整合结果。

当前核心业务对象定义为：

- `TestContext`

`TestContext` 是系统中所有状态判断、规则计算、策略决策的统一输入对象。

### 5.5 Event

事件，用于描述关键状态变化，是试验过程结构化表达的主骨架。

示例：

- `STABLE_REACHED`
- `RECORD_TRIGGERED`
- `CONTROL_SENT`

### 5.6 Action

动作，用于表达系统行为及其执行意图，是决策链路的输出数据。

示例：

- 参数调整
- 工况切换
- 记录控制

### 5.7 Result

结果，用于沉淀可复用的业务产物和阶段性结论。

示例：

- 稳定段结果
- DOE 结果
- 边界结果

### 5.8 Feature

特征，用于模型训练、策略评估、智能优化和高阶分析。

---

## 6. `TestContext` 的项目地位

`TestContext` 是当前系统设计中的核心业务对象，承担统一输入职责。

其地位可归纳为：

- 它不是单一来源数据
- 它是多源融合后的统一上下文对象
- 它是所有计算与决策的唯一直接输入
- 它连接数据链路与决策链路

后续模块设计应围绕 `TestContext` 的形成、消费、回流和追溯展开。

---

## 7. 建模方法

系统采用三层建模结构。

### 7.1 概念实体

概念实体用于表达业务语义，不落入字段或数据库实现细节。

典型示例：

- `Event`
- `Result`
- `TestContext`

### 7.2 逻辑实体

逻辑实体用于表达数据对象，包含可用于系统设计和接口设计的数据结构。

典型示例：

- `MeasurementInput`
- `StandardDataPoint`
- `Event`
- `Result`

### 7.3 物理实体

物理实体用于表达数据库落地结构。

典型示例：

- `t_event`
- `t_result`
- `t_measurement_raw`

---

## 8. 分层建模原则

所有建模工作必须遵守以下原则：

1. 只表达本层职责
2. 不跨层表达语义
3. 不提前引入后续层信息

### 8.1 各层职责边界

数据产生端：

- 只表达来源和输入
- 不表达业务关系

数据融合层：

- 才允许多源关联
- 才允许形成统一业务对象

数据计算层：

- 才允许表达判断逻辑
- 才允许输出计算结论和状态结果

该原则是后续所有实体设计、字段设计和关系设计的硬约束。

---

## 9. 关系建模规范

所有关系必须遵守以下规范。

### 9.1 关系命名

- 使用中文名称
- 使用中文描述

命名示例：

- 产生
- 关联
- 触发
- 形成

### 9.2 基数规则

起始端允许取值：

- 一
- 零或一

结束端允许取值：

- 零或多
- 一或多
- 一
- 零或一

### 9.3 关系类型

标识关系：

- 子实体的存在完全依赖于父实体
- 子实体主键包含父实体主键

非标识关系：

- 子实体与父实体存在关联
- 子实体存在不完全依赖于父实体
- 子实体主键不包含父实体主键

---

## 10. 建模输出规范

每一层建模必须输出以下内容。

### 10.1 实体输出

- 概念实体
- 逻辑实体（含字段）
- 物理实体

### 10.2 模型图输出

必须输出：

- 概念模型图
- 逻辑模型图
- 物理模型图

### 10.3 模型图约束

概念图：

- 只使用概念实体

逻辑图：

- 只使用逻辑实体

物理图：

- 只使用物理实体

补充约束：

- 允许跨分类引用同类型实体
- 不允许在同一张图中混用不同层级实体

---

## 11. 设计约束

后续设计必须持续遵守以下约束：

- 不使用模糊描述
- 不跨层表达逻辑
- 不提前语义化字段
- 不构造无意义关系
- 不重复已有结构

这些约束用于保证模型长期可维护、可扩展、可落地。

---

## 12. 建模工作方式

后续采用按模块逐步设计的方式推进。

单次设计一个模块或一个分类，标准步骤如下：

1. 明确当前层职责
2. 定义概念实体
3. 定义逻辑实体（含字段）
4. 定义物理实体
5. 输出三张模型图
6. 校验是否违反规则

该流程是后续协作和迭代设计的默认工作流。

## 14. 后续设计默认输入

后续任何模块设计都默认以本文件作为背景输入，并基于以下共识继续展开：

- 系统属于工业数据闭环系统，而不是单纯采集系统
- `TestContext` 是统一输入核心对象
- 数据链路与决策链路必须同时考虑
- 事件、动作、结果必须纳入统一数据体系
- 建模必须严格遵守概念层、逻辑层、物理层分离原则
- 每次只针对一个模块完成完整三层设计

## 15. 暂定术语表

为保证后续表达一致，当前暂定使用以下术语。

| 中文术语 | 英文术语 | 当前含义 |
| --- | --- | --- |
| 原始数据 | Raw Data | 保持原始输入形态的数据 |
| 标准数据 | Standard Data | 经过统一格式化后的数据 |
| 上下文数据 | Context Data | 提供业务语义的数据 |
| 融合对象 | Business Object | 多源融合后的统一业务对象 |
| 试验上下文 | TestContext | 系统统一输入核心对象 |
| 事件 | Event | 描述关键状态变化的数据对象 |
| 动作 | Action | 描述系统行为的数据对象 |
| 结果 | Result | 业务产物和阶段结论 |
| 特征 | Feature | 用于训练和优化的数据 |

该术语表后续可继续扩展，但现阶段作为默认表达基线使用。
