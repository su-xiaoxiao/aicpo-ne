# 首批模块拆分清单

## 1. 文档目的

本文档用于定义当前项目首批建模模块、模块职责、上下游关系和设计推进顺序，作为后续逐模块展开设计的工作清单。

## 2. 模块范围

本文档覆盖总体链路下的一级模块拆分。

本文档不覆盖：

- 模块内部字段设计
- 模块内部表结构设计
- 模块内部详细事件类型设计

## 3. 上游输入

本文档以上游输入为基础形成：

- 项目背景基线
- 总体链路设计
- 建模治理规则

## 4. 下游输出

本文档将作为以下工作的直接输入：

- 模块级概念设计
- 模块级逻辑设计
- 模块级物理设计
- 模块级模型图绘制
- 模块级评审

## 5. 模块职责

- 作为后续模块设计的拆分基线
- 作为模块边界判定依据
- 作为设计推进顺序的计划依据

## 6. 核心对象

当前模块拆分围绕以下核心对象展开：

- Raw Data
- Standard Data
- Context Data
- `TestContext`
- `Event`
- `Action`
- `Result`
- `Feature`

## 7. 关键关系

- 数据产生模块 产生 数据接入模块输入
- 数据接入模块 形成 原始数据接入结果
- 数据标准模块 形成 标准数据
- 数据融合模块 形成 `TestContext`
- 数据计算模块 形成 计算结论与特征
- 事件动作模块 触发 `Event` 并形成 `Action`
- 结果沉淀模块 形成 `Result`
- 数据存储模块 记录 全链路数据对象
- 数据服务模块 提供 查询与服务输出
- 数据管理模块 管理 口径、字典、质量与版本

## 8. 分层设计要求

首批模块拆分只表达模块边界与职责，不提前进入字段级和表级设计。

后续每个模块都必须补齐：

- 概念层设计
- 逻辑层设计
- 物理层设计
- 对应模型图
- 评审记录

## 9. 交付清单

- 模块清单
- 模块职责说明
- 模块输入输出说明
- 模块依赖顺序
- 模块设计优先级

## 10. 首批模块清单

| 模块编码 | 模块名称 | 模块定位 | 核心输出 |
| --- | --- | --- | --- |
| M01 | 数据产生模块 | 描述数据来源边界 | 原始外部输入、原始内部输入 |
| M02 | 数据接入模块 | 承接多源输入接入 | 已接入原始数据 |
| M03 | 数据标准模块 | 统一结构与编码 | Standard Data、Context Data |
| M04 | 数据融合模块 | 形成统一业务对象 | `TestContext` |
| M05 | 数据计算模块 | 进行判断和计算 | 计算结论、Feature |
| M06 | 事件动作模块 | 结构化过程与动作 | `Event`、`Action` |
| M07 | 结果沉淀模块 | 沉淀业务结果 | `Result` |
| M08 | 数据存储模块 | 承接全链路存储 | 可追溯数据资产 |
| M09 | 数据服务模块 | 对外提供数据能力 | 查询结果、服务对象 |
| M10 | 数据管理模块 | 提供治理与标准 | 元数据、质量规则、字典 |

## 11. 模块定义

### 11.1 M01 数据产生模块

职责：

- 定义数据来源边界
- 区分外部输入和内部输入
- 识别原始数据产生位置

输入：

- 设备侧数据源
- 外部系统侧数据源
- 内部控制与执行输出

输出：

- 原始测量数据
- 原始反馈数据
- 原始动作反馈数据

### 11.2 M02 数据接入模块

职责：

- 接入多源输入
- 保留接入轨迹
- 统一接入过程边界

输入：

- M01 输出

输出：

- 已接入原始数据

### 11.3 M03 数据标准模块

职责：

- 标准化数据结构
- 标准化时间和编码
- 提取可复用上下文数据

输入：

- M02 输出

输出：

- Standard Data
- Context Data

### 11.4 M04 数据融合模块

职责：

- 融合标准数据和上下文数据
- 形成统一输入对象

输入：

- Standard Data
- Context Data
- 当前控制状态

输出：

- `TestContext`

### 11.5 M05 数据计算模块

职责：

- 对 `TestContext` 进行状态判断
- 形成稳定性、窗口、特征等计算结果

输入：

- `TestContext`

输出：

- 状态判断结果
- 特征数据

### 11.6 M06 事件动作模块

职责：

- 将关键状态变化表达为 `Event`
- 将控制决策表达为 `Action`
- 记录动作执行过程

输入：

- 计算结论
- 决策输出
- 执行反馈

输出：

- `Event`
- `Action`

### 11.7 M07 结果沉淀模块

职责：

- 沉淀稳定段结果
- 沉淀 DOE 结果
- 沉淀边界结果和其他业务结果

输入：

- `Event`
- `Action`
- 计算结论
- 特征数据

输出：

- `Result`

### 11.8 M08 数据存储模块

职责：

- 管理全链路数据持久化
- 建立追溯基础

输入：

- M01 至 M07 输出对象

输出：

- 可持久化数据资产

### 11.9 M09 数据服务模块

职责：

- 对内对外提供数据访问服务
- 向分析、算法、评审场景提供数据支撑

输入：

- M08 存储结果

输出：

- 服务化查询结果

### 11.10 M10 数据管理模块

职责：

- 管理标准、字典、元数据、版本、质量规则
- 维护跨模块的一致性约束

输入：

- 全链路对象定义
- 治理规则

输出：

- 数据治理结果

## 12. 模块依赖顺序

首要依赖顺序如下：

1. M01 数据产生模块
2. M02 数据接入模块
3. M03 数据标准模块
4. M04 数据融合模块
5. M05 数据计算模块
6. M06 事件动作模块
7. M07 结果沉淀模块
8. M08 数据存储模块
9. M09 数据服务模块
10. M10 数据管理模块

其中：

- M10 为横向治理模块，可与各模块并行补充
- M08、M09 在详细建模时可能按对象类型拆分子模块

## 13. 模块设计优先级

建议后续优先设计顺序如下：

1. M03 数据标准模块
2. M04 数据融合模块
3. M05 数据计算模块
4. M06 事件动作模块
5. M07 结果沉淀模块
6. M02 数据接入模块
7. M08 数据存储模块
8. M09 数据服务模块
9. M10 数据管理模块
10. M01 数据产生模块

优先原因：

- 数据标准、融合、计算决定核心对象边界
- `TestContext` 的定义会反向约束接入、事件、结果和存储设计
- 数据来源边界可在后续结合实际设备和系统进一步细化

## 14. 待确认事项

- 数据管理模块是否在首批建模中单独成册
- 数据服务模块是否需要拆成内部服务与外部服务两个模块
- 结果沉淀模块是否需要拆分为试验结果、执行结果、训练样本三个子模块
- 数据计算模块是否需要拆出“状态判断”和“特征计算”两个模块
